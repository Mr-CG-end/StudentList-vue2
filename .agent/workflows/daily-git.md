# 日常开发工作流程

## 📝 标准流程

### 1. 启动开发服务器(每天开始工作时执行一次)

```bash
npm run serve
```

- 📌 服务会在 `http://localhost:8080` 启动
- 🔄 修改代码后浏览器会**自动刷新**

---

### 2. 日常开发循环

```
修改 .vue 文件 → 保存 → 浏览器自动刷新 → 继续修改...
```

---

### 3. 提交代码到本地仓库(每完成一个功能)

```bash
# 查看修改了哪些文件
git status

# 添加所有修改到暂存区
git add .

# 提交到本地仓库(附上功能描述)
git commit -m "feat: 添加学生搜索功能"
```

**Commit 消息类型参考**:

- `feat:` 新功能
- `fix:` 修复 bug
- `style:` 样式调整
- `refactor:` 代码重构
- `docs:` 文档更新

---

### 4. 推送到 GitHub(备份代码)

```bash
# 推送到远程仓库
git push
```

- ✅ 执行后 GitHub 上的代码会立即更新
- 📦 这样你的代码就有了**云端备份**

---

## 🚀 快速参考

| 场景                 | 命令                     |
| -------------------- | ------------------------ |
| 查看文件修改状态     | `git status`             |
| 添加所有修改         | `git add .`              |
| 提交到本地           | `git commit -m "描述"`   |
| 推送到 GitHub        | `git push`               |
| 查看提交历史         | `git log --oneline -10`  |
| 撤销工作区修改(危险) | `git checkout -- 文件名` |

---

## ⏰ 推荐节奏

| 时间                  | 操作                           |
| --------------------- | ------------------------------ |
| **每天开始**          | `npm run serve` 启动开发服务器 |
| **开发过程**          | 随时修改保存,浏览器实时预览    |
| **完成一个功能**      | `git add .` → `git commit`     |
| **下班前 / 每天结束** | `git push` 推送到 GitHub       |

---

## 💡 最佳实践

1. **小步提交**: 完成一个独立功能就 commit,不要积累太多修改
2. **清晰描述**: commit 消息要说明做了什么,方便日后查找
3. **每天推送**: 至少每天结束工作前 push 一次,防止代码丢失
4. **先拉取再推送**: 如果多设备开发,`git push` 前先执行 `git pull`

---

## 🆘 常见问题

### Q1: `git push` 提示 "rejected"

**原因**: 远程仓库有新提交,本地落后了
**解决**:

```bash
git pull --rebase  # 拉取远程更新
git push           # 再次推送
```

### Q2: 不小心修改了不想提交的文件

```bash
git checkout -- 文件名  # 恢复该文件到上次提交状态
```

### Q3: commit 消息写错了

```bash
git commit --amend -m "正确的消息"  # 修改最后一次commit
```
